# Native Method

Version: v1.6

This repository now separates the native method from the classical translation layer.

The native method does not begin from the usual Collatz question:

Does every positive integer eventually reach 1?

That question is a standard mathematical framing.

The native method begins from a different object:

What happens to expansion, discharge, debt, shadow, regeneration, and compensation inside the trajectory?

The native method treats these as primary structural objects.

## Core position

The previous computational scans are valid evidence layers.

They are not the native theory itself.

They were produced after partial translation into measurable finite protocols.

The native method must be exposed before it is translated back into standard mathematics.

## Boundary

Native method:

- observes odd-block behavior
- tracks expansion and discharge
- measures debt accumulation
- studies shadow length
- detects regeneration
- tests compensation
- searches for obstruction structures

Classical translation:

- restates native structures in conventional mathematical language
- defines lemmas
- defines inequalities
- attempts formal proof
- communicates results to standard mathematical audiences

The order matters.

The repository must not collapse the native method into standard notation too early.

## Native sequence

The correct order is:

1. Native primitives
2. Native dynamics
3. Native obstruction question
4. Native closure criterion
5. Computational evidence
6. Standard mathematical translation

The incorrect order is:

1. Standard Collatz statement
2. Classical notation
3. Computation
4. Native interpretation

That order hides the native structure instead of exposing it.

## Primary native question

Can a debt structure exist that never generates compensating discharge?

If the answer is no, the native structure has a closure mechanism.

If the answer is yes, the native method should expose the obstruction candidate.

This is not yet claimed as a proof.

It is the research path.
